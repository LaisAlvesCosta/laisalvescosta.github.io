(available at <https://flusp.ime.usp.br/kernel/modules-intro/>)


In this post, I document my experience with **FLUSP Tutorial 3**, which focuses on Linux kernel build configuration and module management. The goal was to create a simple module (`simple_mod`), understand how Kconfig symbols work, and learn how to dynamically load and unload code in the kernel.

Below, I detail the steps followed, the necessary corrections to the original tutorial, and the connection issues I encountered.

During execution, I noticed a few errors in the original material that might confuse other developers:

*   **Kernel Image Path:** In the installation step, the tutorial suggests the image generated by `virt` would be at `${IIO_TREE}/arch/arm/boot/Image`. However, for the architecture we are using, the correct path is **`${IIO_TREE}/arch/arm64/boot/Image`**.
*   **Unmounting the Rootfs:** When executing the command to unmount the file system, the correct command is `sudo guestunmount "${VM_DIR}/arm64_rootfs"`, rather than the generic mount point suggested previously.

Also, I initially confused `vda2` with `sda2`. After reviewing Tutorial 1, I confirmed that the rootfs is `sda2`.

One of the biggest challenges was an unexpected change in the virtual machine's IP, which prevented the use of `kw ssh`. To resolve this, I used the following command to identify the new IP address: `sudo virsh net-dhcp-leases default`. Since `kw` and the VM were on different IPs, I had to locate where the old configuration was stored using `grep` and manually update the **`/home/lk_dev/iio/.kw/remote.config`** file with the new IP.

To enable the `simple_mod` module, I used the **menuconfig** interface:
1.  Access the menu: `make -C "$IIO_TREE" menuconfig`.
2.  Search for `simple_mod` (using `/`) and select the **`m`** option to enable it as a module.
3.  Then, I performed a clean build using `kw`:
    *   `$KW_DIR/kw build --clean`
    *   `$KW_DIR/kw build`

After completing these steps, I repeated step 3 to properly build the simple_mod_part module as well.

Then, booting the VM, I confirmed the kernel version and module information:

```bash
root@localhost:~# uname --all
Linux localhost 7.0.0-rc4VM-laistar+ #2 SMP PREEMPT Tue Mar 24 20:05:51 -03 2026 aarch64 GNU/Linux

root@localhost:~# modinfo simple_mod
filename:       /lib/modules/7.0.0-rc4VM-laistar+/kernel/drivers/misc/simple_mod.ko
license:        GPL
vermagic:       7.0.0-rc4VM-laistar+ SMP preempt mod_unload aarch64
```

When loading the module, the kernel records log messages that can be viewed with `dmesg`. In my test, inserting and removing the module resulted in the expected output:

*   **Load:** `[  185.609413] Hello world`
*   **Removal:** `[  277.006912] Goodbye world`

This tutorial is fundamental for understanding Linux modularity, allowing for the addition of features without recompiling the entire kernel or restarting the system. One of the most valuable lessons was understanding the difference between loading tools. While `insmod` requires a direct path to a .ko file and ignores dependencies, `modprobe` is much more efficient. In scenarios where a module exports functions called by others (such as the example of simple_mod being used by simple_mod_part), the developer does not need to manually load each part; modprobe resolves these links automatically. 

Note: the full compilation via `kw build` took approximately **17 minutes** in my environment.
